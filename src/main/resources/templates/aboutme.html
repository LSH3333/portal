[지인의 패션 브랜드 쇼핑몰 개발]

https://mavi-karga.com/

지인의 패션 브랜드 쇼핑몰 개발을 했습니다.
현재 서버,클라이언트는 대부분 완성된 상태이고 일부 제품 설명, 컨텐츠만 채워 넣으면 되는 상태입니다.

지인이 직접 상품 추가, 수정 등 관리가 가능하도록 프로그래밍을 모르는 관리자가 손쉽게 상품 추가, 수정, 주문조회, 회원조회를 할 수 있도록 어드민 페이지를 만들었습니다.
결제는 포트원 api 를 사용했고, 결제 검증은 포트원에서 보내주는 웹훅을 이용해 안전하게 검증하도록 했습니다.
결제 시스템은 현재 PG사와 계약이 완료되면 테스트 결제에서 실제 결제가 가능하도록 변경 예정입니다.

실제 쇼핑몰 서비스를 위해서는 개인정보 취급방침, 이용약관, 배송 방침 등 지켜야 할 일들이 많았고 특히 개인정보 보관이 엄격해 3개월에 한 번 DB 에서 개인정보는 자동 제거하도록 했습니다.
개인정보 보호와 데이터 무결성을 지키기 위해 테이블 자체를 지우는 hard deletion 이 아닌 내용만 지우는 soft deletion 으로 처리해 혹여나 연관관계에 문제가 없도록 했습니다.

[여러 개의 옵션이 있는 상품의 도메인 설계]

쇼핑몰 개발 초기에는 상품의 선택 옵션이 사이즈 선택 밖에 없었지만, 이후 일부 상품은 색상의 선택을 갖게 됐습니다.
상품이 상품 사이즈 옵션만을 갖을 때는 상품 테이블과, 상품 사이즈 테이블 사이에 일대다 관계를 만드는 것으로 충분했습니다.
하지만 색상 이라는 옵션이 추가되면서 도메인 구조 자체의 변경이 불가피하게 됐습니다.
문제 해결을 위해 키-값 테이블을 만드는 등 다른 해결방법을 생각해봤지만 여기서는 다른 방법을 사용했습니다.

상품 사이즈, 색상을 갖는 상품옵션 테이블을 만들었고, 상품 테이블과 일대다 관계를 맺었습니다.
이렇게 설계 할 경우 문제가 있었는데 상품 사이즈와 색상 옵션의 조합에 비례하여 중복 데이터가 DB에 저장된다는 문제입니다.
하지만 지인과 상담결과 이 쇼핑몰의 경우 거의 모든 상품은 사이즈 옵션만 존재하고 극히 일부 상품 (현재로서는 1~2개의 상품 정도만) 만 색상 옵션도 갖기 때문에 충분히 적용 가능하다고 판단했습니다.

[결제 시스템, 결제금액 사후검증 웹훅 처리]

mavi-karga 쇼핑몰의 결제 사후검증 방식은 처음에 사용자가 입력한 정보(이메일,배송지 등)와 결제관련 정보를 클라이언트에서 서버로 한번에 보내도록 구현하였었습니다.
이후 포트원에서 보내주는 웹훅을 이용해 결제검증을 하도록 변경이 필요했는데, 사용자가 입력한 정보를 서버로 보내는 요청과, 웹훅의 결제 검증 로직을 분리해야 했고 HTTP는 상태를 저장하지 않기 때문에 두 개의 서로 다른 요청을 처리하는 어려움이 있었습니다.
웹훅은 포트원 서버에서 보내고, 사용자의 정보는 사용자의 클라이언트에서 보내기 때문에 세션도 서로 달라 세션 사용도 어려웠습니다.

고민 끝에 해결한 방법은 포트원 웹훅에서 보내주는 merchant_uid 라는 고유 값을 식별자로 사용했습니다.
먼저 사용자의 클라이언트에서 사용자 정보를 받아서 데이터를 DB에 저장하고 이때 merchant_uid 도 같이 저장합니다.
이후 포트원에서 보내는 웹훅을 받아 merchant_uid 값으로 DB에서 데이터를 가져와서 결제 검증 후 결제정보를 완성해 DB에 저장하도록 해서 문제를 해결했습니다.

[부족한 부분을 개선하기 위한 버전 2 제작]

Instaweb-v1

Instaweb-v2

만들어본 첫 번째 웹 프로그램은 Instaweb 이 었습니다.
처음에는 에디터, OAuth2 로그인 등 이 프로젝트에서 필요한 기능들을 최대한 프레임워크에 의존하지 않고 직접 구현해보고 싶었습니다.

이후 Instaweb 을 만들면서 개선하고 싶었던 부분을 고친 버전 2 를 만들었습니다.
최대한 프레임워크에 의존하지 않고 구현해본 버전 1과 다르게 버전 2 에서는 OAuth2 로그인을 위한 Spring Security와 에디터는 ckeditor5 api 를 활용했습니다.

버전 1 은 heroku로 배포한 반면 버전 2 는 Docker 를 이용해 aws ec2 배포 했습니다.
또한 새로운 도메인을 구입하지 않고 보유중이던 sehyun-portal.com 의 네임서버를 AWS Route53를 가리키도록 해서 서브도메인을 활용했습니다.
