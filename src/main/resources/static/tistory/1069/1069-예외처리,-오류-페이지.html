예외처리, 오류 페이지

Web

2023-02-17 17:30:56

 

 

웹 어플리케이션에서 예외가 발생했을때 어플리케이션 내에서 처리해주면 된다.

그런데 만약 어플리케이션에서 처리되지 못하면 톰캣같은 Web Application Server 까지 예외가 전달된다.

이 경우 WAS 는 서버에서 예외가 발생한것으로 감지하고 500 에러를 보낸다.

 

 

 

그런데 이렇게 어떤 경로에 대하여 예외가 발생했을때 사용자에게 그냥 500, 404 에러 페이지를 보여줄수는 없다.

예외가 발생했을때 인터셉터나 필터로 직접 구현한 예외 페이지 뷰를 보여주도록 할 수 있다.

 

 

그러나 이런 과정들은 모두 스프링 부트에서 제공한다. 

 

 

 

스프링 부트의 오류 페이지 

- 스프링 부트는 ErrorPage를 자동으로 등록하고 "/error" 경로를 기본 오류 페이지로 설정한다.

- 서블릿 밖으로 예외가 발생하거나 response.sendError() 가 호출되면 모든 오류는 "/error" 를 호출한다.

- BasicErrorController 스프링 컨트롤러를 자동 등록한다. 여기서 "/error" 를 매핑해 처리한다. 

 

즉 오류가 발생하면 "/error" 경로를 요청하게 된다.

따라서 우리는 해당 경로에 뷰만 구현해놓으면 된다. 

 

 

뷰 선택 우선 순위 

BasicErrorController 의 처리 순서.

 

1. 뷰 템플릿 

  • resources/templates/error/500.html
  • resources/templates/error/5xx.html

2. 정적 리소스 

  • resources/static/error/400.html
  • resources/static/error/404.html
  • resources/static/error/4xx.html

3. 적용 대상이 없을때 뷰 이름 (error) 

  • resources/templates/error.html

 

뷰 템플릿이 정적 리소스보다 우선 순위가 높고 404,500 처럼 구체적인 것이 4xx 처럼 덜 구체적인 것 보다 우선순위가 높다.

4xx 는 400 번대 오류를 처리한다.
